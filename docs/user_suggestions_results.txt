## Experiment 2: Embedding Models (MiniLM vs Nomic)
**Date**: 2026-02-06
**Result**: ✅ **Nomic Wins**.
*   **MiniLM**: 313ms, 384 dim, 512 context. (Too short for papers).
*   **Nomic**: 1481ms, 768 dim, **8192 context**.

**Architecture Decision**:
Since "time doesn't matter" for ingestion, **Nomic-Embed-Text-v1.5** is the superior choice. It allows us to embed the *entire abstract + introduction + conclusion* as a single vector, capturing the full "thesis" of a paper rather than fragmented chunks.

## 4. Storage & Retrieval Upgrades (Recommendation)
**User Request**: "Better open source alternative to chromaDB... make data storage meaningful."

### Vector Database: **LanceDB** 🚀
*   **Why**: It is the "SQLite of Vector DBs".
    *   **Architecture**: Serverless, runs in-process (like Chroma) but uses the ultra-fast **Lance** columnar format.
    *   **Performance**: Can handle billions of vectors on local disk without eating RAM. 100x faster filtering than Chroma.
    *   **Interoperability**: Native integration with Pandas/Arrow (great for data science analysis).

### Knowledge Graph: **KùzuDB** (or NetworkX) 🕸️
*   **Why**: You asked for "concept mapping".
*   **KùzuDB**: An in-process, highly optimized Graph DB designed to work with vector search. It's the "DuckDB of Graphs".
*   **Plan**:
    1.  Use Qwen to extract `(Entity)` -> `[Relation]` -> `(Entity)`.
    2.  Store in Kùzu/LanceDB.
    3.  **GraphRAG**: When querying, we traverse the graph to find "Hidden Connections" before doing the vector search.

### Final Stack recommendation ("TigerStack 2.0"):
*   **Ingestion**: `SmartCrawler` (Qwen VLM/JSON) + `DeepDistiller` (Qwen Summary).
*   **Embedding**: `Nomic-Embed-Text-v1.5` (8k context).
*   **Storage**: `LanceDB` (Vectors) + `Kùzu` (Graph).
*   **Inference**: `Ollama` (Qwen 2.5).

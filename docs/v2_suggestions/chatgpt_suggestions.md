TigerStack 2.0 Architecture Review

Core Architecture Critique

The current “Two-Lobe Brain” design sends each query in parallel to LanceDB for vector similarity and to the TigerBrain graph for concept/faculty relations ￼.  This hybrid RAG approach indeed unites semantic search with explicit knowledge, which is a powerful pattern: vector search finds “what sounds like this,” while graph traversal finds “what connects to this” ￼.  However, this parallel strategy has trade-offs.  In practice, parallel retrieval is fast and resilient (both lobes run concurrently), but it risks redundant or overlapping results and bloated prompts.  As noted in the plan, sending both full vector and graph contexts can bloat LLM input and complicate result merging.  By contrast, a sequential approach (graph then vector) could narrow the vector search to only graph-relevant documents.  Industry guidance suggests sequential (or cascading) retrieval is often simpler and sufficient unless ultra-low latency is needed ￼.  For example, DataRobot observes that sequential retrieval (vector-first, then graph) is “most straightforward… If it’s working without significant cost to latency or accuracy, most organizations should stick with it” ￼.  The trade-off is higher latency and brittleness (if entity extraction fails, nothing is retrieved), but the benefit is a more focused, smaller LLM context.

An even smarter hybrid pattern is adaptive query routing: classify the query and choose the best strategy.  The architecture discussion itself suggests routing “entity-focused” queries (e.g. “Who works on X?”) to the graph first, and concept questions to the vector search.  This aligns with advice from hybrid RAG research: use graph-first for narrowly scoped relational questions and vector-first for broad or exploratory queries ￼.  Over time, a learned classifier or simple rules could improve this routing.  Another enhancement is summarization-first retrieval.  Rather than feeding hundreds of raw chunks to the LLM, one can first compress or summarize the vector-retrieved passages, then merge with the graph context.  Hierarchical RAG techniques recommend “compress or summarize retrieved passages into a single coherent context window” before generation ￼.  In fact, the decision log itself notes “Two-Stage Synthesis: Summarize vector results first, then combine with graph” as a mitigation for token limits ￼.  Combining these ideas, a potential pipeline could be: extract entities → graph traverse → vector retrieve relevant docs → summarize those docs → LLM answer.  This would yield smaller, higher-quality context at the cost of extra steps.  Given the single-user, accuracy-prioritized constraints, the added complexity and latency may be acceptable if it improves answer coherence.

In summary, the two-lobe parallel design ensures comprehensive coverage but can overwhelm the LLM with context and redundant facts.  A sequential or adaptive strategy – graph-guided filtering of vectors, or query-type routing – could focus retrieval and reduce prompt size ￼ ￼.  Likewise, applying context condensation (e.g. summarization or pruning) can mitigate LLM overload ￼ ￼.  These refinements leverage known hybrid RAG patterns: combine dense and sparse (graph) search, optionally rerank/summarize, and adapt based on query type ￼ ￼.

Scalability & Performance Risks

Several scaling risks emerge when growing beyond the current 1,000-paper graph.  Most pressing is graph performance: TigerBrain is built with NetworkX (in-memory). The prototype graph already has ~45,000 nodes and 46,000 edges ￼.  Extrapolating, 10× more papers could mean hundreds of thousands of nodes.  NetworkX is known to struggle past ~100k nodes ￼.  In practice, large in-memory graphs incur heavy RAM use and slow traversal.  The architecture notes this as “Risk 1”, suggesting migrating to a graph database (Neo4j/KuzuDB) or adding indices.  Those mitigations will be needed: e.g. indexing common queries and caching subgraphs (e.g. “all papers by Prof. X”).  Without it, graph queries may exceed the latency budget and even cause memory swaps.

Another risk is data freshness and ingestion consistency.  The plan envisions an automated “Watcher” pipeline that distills new PDFs and updates both TigerBrain and LanceDB ￼.  In large-scale systems, ingestion pipelines run asynchronously to avoid blocking queries ￼.  If a user queries while new papers are mid-index, the knowledge base can be inconsistent.  As one analysis warns: if an ingestion job partially completes, retrieval could return a mix of old and new content (“contradictory answers”) ￼ ￼.  To mitigate this, production systems use versioning or atomic updates: only swap in the new index when fully ready ￼ ￼, or label data with timestamps to signal staleness.  We should plan similar safeguards (locks or versioned indices) so that graph updates don’t corrupt ongoing queries ￼.

A third concern is LLM context and latency.  Combining two retrieval lobes risks exceeding token limits or making the prompt hard for the model to handle.  While Qwen-2.5-32B supports an enormous context (over 100K tokens) ￼, pragmatic output and inference limits (e.g. 8K tokens generated) remain.  The document warns that “combined context (Vector + Graph) might exceed LLM token limits”.  To avoid this, we may need context trimming: e.g. drop lower-ranked chunks, truncate long passages, or summarize intermediate results ￼ ￼.  Qwen’s large context helps, but prompt assembly time and generation latency on a 32B model could be significant.  We must ensure the GPU infrastructure can handle worst-case context windows without excessive delay.

Finally, on the vector DB side, 10k–100k documents is trivial for LanceDB or FAISS-like indexes, so we expect minimal issues there.  The bigger risk is GPU and memory load from the 32B LLM on long prompts.  The parallel design reduces wait time by overlapping retrieval, but the single-user allowance means we could also afford sequential steps if needed.  Overall, the primary scaling pressure is on the graph component and the LLM context assembly, not on vector search.

Missing Production Components

To harden the system for production, several standard RAG components and practices should be added.  Key missing pieces include:
	•	Caching (Semantic & Prompt): Implement a caching layer for frequent or similar queries.  Production RAG architectures use semantic caching – storing embeddings of queries and responses so that if a new query is nearly identical to a past one, the cached answer can be reused ￼ ￼.  Even a simple first-layer cache of recent Q&A or vector results will greatly reduce redundant work.  Prompt caching (if supported by the LLM) can also cut repeated costs ￼.
	•	Re-Ranking / Two-Stage Retrieval: After initial retrieval (vector and/or graph), a secondary reranking step will improve answer quality.  A common pattern is: do a fast bi-encoder (LanceDB) search to get ~100 candidates, then use a slow cross-encoder or specialized model to re-score those ￼ ￼.  This cross-encoder stage models query-chunk interactions for higher precision.  The plan’s “HybridRetriever” could be extended to run a reranker on the combined result set ￼.  Empirical results show 20–40% precision gains from such two-stage pipelines ￼.
	•	Query Expansion/Rewriting: Enhance the retriever with query reformulation.  For example, use the LLM to expand or clarify the user’s question (extracting keywords, synonyms, context) before search ￼ ￼.  The plan already suggests extracting entities, but we can also expand acronyms or add detail (“ML” → “Machine Learning”).  More advanced techniques like HyDE (generate a hypothetical answer then search its embedding) could also be evaluated ￼.  These steps increase recall and make both vector and graph search more robust.
	•	Query Classification & Smart Routing: Add a module to classify queries by type (factual, open-ended, entity lookup, etc.) and route them accordingly.  This dovetails with the adaptive hybrid idea.  As one review notes, “Production systems implement query classification: simple queries go to cheap models… complex queries requiring reasoning go to expensive models” ￼.  Here, classification could choose which retriever first or even which LLM to invoke (we only have one model, but could skip graph search for certain queries, or throttle LLM verbosity).  At minimum, rules or a trained classifier can gate whether TigerBrain is used.
	•	Asynchronous Processing & Pipelines: The ingestion of new papers must run as a reliable background pipeline (e.g. worker queue) rather than inline with queries ￼.  The plan’s “Watcher” should manage failures, resume partially processed files, detect duplicates, and do blue-green updates to indices ￼ ￼.  This ensures the query service is always responsive, even during large ingests.
	•	Observability & Logging: Instrument every component with logging and metrics.  We should track latency of vector and graph queries, cache hit rates, LLM token usage, and answer quality over time ￼ ￼.  Critically, log everything: which documents/subgraphs were retrieved, the assembled prompt, and the final answer ￼.  These logs (and distributed traces) are essential for debugging when answers go awry.  In production RAG, test suites with baseline queries are also used to catch regressions.
	•	Fallback Logic: Define graceful degradation if a component fails.  For example, if TigerBrain is down or the entity node isn’t found, the system should still return vector-based answers ￼.  The architecture discussion explicitly calls out a fallback to vector-only retrieval ￼.  Similarly, if the LLM run fails (OOM or timeout), catch the error and return a partial answer or a default message rather than crashing the service.
	•	Index and Model Versioning: Plan for re-indexing and model updates.  Every time the embedding model changes, all chunks should be re-embedded or versioned, else the vector space drifts ￼.  Maintain version tags for TigerBrain snapshots as well, so queries are consistent.  Also, ensure no stale cache (invalidate cache on data update) ￼.

Each of these elements (caching, reranking, expansion, logging, etc.) moves the prototype toward a robust, maintainable system.  Many are noted in the decision log’s “Missing Pieces” list ￼, which we should implement in Phase 2.

Alignment with Constraints

The proposed design generally fits the stated constraints: it is fully on-premise and privacy-first, and optimized for a single user who tolerates latency.  All data (faculty pages, PDFs) and services (LanceDB, TigerBrain, Qwen model) run locally, ensuring no data leaves the infrastructure.  The use of the Qwen-2.5 32B LLM (fine-tuned for long context) is consistent with a privacy-focused, on-prem deployment ￼.  Its huge 128K context window means we are unlikely to hit token limits in normal use, reducing one typical concern of hybrid prompts.  However, we must still plan GPU resources: a 32B model needs substantial VRAM and will generate tokens slowly, so multi-second latencies are expected.  This is acceptable under “accuracy over speed” goals, but we should still monitor throughput if usage grows.

Single-user tolerance allows us to favor thoroughness.  For example, the parallel design is optimized for speed (searches run concurrently), but since we only support one user at a time, we could also afford sequential or multi-stage strategies without blocking other users.  In fact, if latency is not critical, a graph-then-vector (or summarization) pipeline may yield better answers.  The architecture plan itself could explicitly note this trade-off: a fully sequential mode (Graph → Vector → LLM) might slow the response by a few hundred milliseconds, but still within acceptable limits, and could improve precision ￼.

Finally, Qwen’s context capacity (131K tokens ￼) means we have more headroom for long answers.  We should still be careful with LLM token usage for cost/performance, but the privacy mandate outweighs cost concerns here.  In sum, the plan’s architecture (local vector + graph + on-prem LLM) is well-aligned with all given constraints.

Recommendations & Next Steps
	1.	Choose Retrieval Workflow: Decide whether to keep fully parallel retrieval or adopt a sequential/hierarchical pipeline.  Given the single-user, accuracy-first context, we recommend prototyping sequential graph→vector or two-stage (summarize + merge) flows and comparing answer quality.  If answers improve without critical latency issues, prefer the sequential/hierarchical pattern ￼ ￼.
	2.	Scale the Graph Backend: Benchmark TigerBrain performance at target scale.  If NetworkX proves too slow or memory-hungry, plan to migrate to a graph DB (Neo4j, KuzuDB, etc.) ￼.  In the meantime, implement node/edge indexing and common subgraph caches to speed up frequent traversals.
	3.	Implement Caching Layers: Build a semantic cache keyed by query embedding (for full Q&A caching) and a prompt cache (for repeated instructions/context) ￼ ￼.  This will cut redundant work and speed up repeated queries (e.g. “Who works on AI?”).
	4.	Add Reranking and Query Rewriting: Integrate a cross-encoder reranker over the top-k retrieval results ￼ ￼.  Also implement LLM-based query rewriting or expansion to improve retrieval, especially for ambiguous queries ￼ ￼.
	5.	Develop Observability: Instrument the pipeline end-to-end.  Log retrieval outputs, assembled prompts, and final answers for each query ￼.  Set up metrics (latency, hits, relevance scores) and run regular test queries to catch regressions.
	6.	Harden Ingestion: Finalize the “Watcher” as an asynchronous worker with error handling.  Use versioned indexes or read-write locks during updates to prevent inconsistent reads ￼ ￼.  Ensure new documents are deduplicated and chunking is consistent.
	7.	Implement Fallbacks: Code logic so that if TigerBrain lookup fails (e.g. missing node), the system still returns vector-only answers ￼.  Similarly, catch and handle errors in LLM inference or LanceDB search to avoid service crashes.

By iterating through these steps—evaluating retrieval patterns, bolstering infrastructure, and adding robustness features—we can move the prototype into a reliable production-ready system. These changes, together with thorough testing and monitoring, will ensure the TigerStack hybrid RAG meets its design goals under real-world scale and usage.
